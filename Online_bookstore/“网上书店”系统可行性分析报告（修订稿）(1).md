**项目名称：** 网上书店管理系统

**项目小组：**ReadStore

------

## 1. 引言

### 1.1 编写目的

本报告旨在对“网上书店”系统的开发进行全面的可行性研究，从技术、经济、操作及法律社会等维度评估项目的可行性与潜在风险。本报告将作为项目立项决策的核心依据，并为后续的系统设计与开发工作提供明确的指导框架。报告同时整合了对计算机辅助软件工程（CASE）工具的调研结果，明确了项目开发将采用的技术支持和协同工具。

### 1.2 项目背景

在互联网与电子商务深度普及的背景下，传统实体书店的经营模式面临巨大挑战。为拓展销售渠道、降低运营成本并提升服务效率，开发一个功能完备、用户友好的网上书店系统成为必然选择。本项目旨在为一个模拟的中小型书店构建一个集图书在线展示、销售、客户服务与后台管理于一体的综合性系统，采用经典的桌面应用程序（C/S）架构，以实践软件工程的全流程。

## 2. 项目概述

### 2.1 系统目标

构建一个安全、稳定、易用的网上书店桌面管理平台。该系统需实现：为顾客（游客与会员）提供便捷的图书浏览、查询、选购与下单体验；为书店管理者提供高效的图书进销存管理、客户关系维护、账目统计及网站内容维护功能，从而全面实现书店业务的信息化与网络化。

### 2.2 核心功能需求

1. **前台客户系统**： **游客**：无需登录即可浏览图书信息、查看书店简介与最新公告。 **会员**： **账户管理**：注册（取得唯一用户名）、登录。 **图书选购**：分级浏览图书、将选定图书加入购物车。 **订单管理**：生成订单、模拟支付流程、查看个人历史订单。 **互动功能**：在网站留言板留言。
2. **后台管理系统**： **图书管理**：对图书信息进行增、删、改、查（CRUD）；实时监控与更新库存数量。 **订单管理**：处理会员订单（确认订单、标记发货）、查看所有销售记录。 **客户管理**：查看已注册会员信息列表；管理与回复客户留言。 **账目管理**：进行简单的销售收入、成本支出统计，生成基础报表。 **网站维护**：发布或更新书店公告、新闻；维护“关于我们”、“书店简介”等静态页面内容。

### 2.3 假设与约束

- **技术约束**：项目核心业务逻辑使用C++语言实现，可视化图形用户界面（GUI）采用C# (Windows Forms / WPF) 开发，在Microsoft Visual Studio集成开发环境下进行。
- **时间约束**：项目需在课程规定的周期内完成全部开发、测试与文档交付工作。
- **资源约束**：项目成功依赖于小组成员的自主学习能力、技术实践效率以及有效的团队协作。无商业预算支持。
- **功能约束**：支付环节采用模拟流程，不接入真实支付网关。

## 3. 可行性分析

### 3.1 技术可行性

- **技术栈成熟度**：拟采用的C++（用于高性能核心逻辑）、C#（用于构建高效的Windows桌面界面）以及Visual Studio IDE，均是业界极为成熟、拥有海量学习资源和社区支持的技术组合，学习曲线相对平缓。
- **功能实现评估**：系统所需功能，如用户认证、数据的增删改查、购物车与订单流程等，均属于典型的商业应用场景，有大量的设计模式与实现案例可供参考。数据持久化层可采用轻量级的SQLite或通用的MySQL数据库，技术实现路径清晰。
- **团队技术储备**：小组成员具备相关的程序设计语言与数据库基础，通过项目驱动的学习，能够掌握必要技术的应用。
- **结论**：在所选技术路线下，无不可逾越的技术障碍，项目**技术可行**。

### 3.2 经济可行性

- **开发成本**：主要为人力成本。本项目为课程实践，开发人员（学生）无薪资支出。所有开发工具（Visual Studio Community版）、版本控制与协作平台（GitHub/Gitee）、设计工具（Draw.io）均可免费使用。
- **运营成本**：系统交付后，可运行于普通的Windows PC或低成本服务器上，无需额外的昂贵硬件或软件授权费用。
- **预期收益**：本项目为非营利性教学项目，主要收益体现在：小组成员获得完整的软件工程实战经验、深入理解C++/C#开发与系统设计、完成课程考核目标。从教育投资回报角度看，价值显著。
- **结论**：项目开发与运营成本极低，教育收益明确，**经济可行**。

### 3.3 操作可行性（运行可行性）

- **用户界面**：计划设计为符合Windows操作习惯的桌面窗口应用程序，布局清晰、导航直观、控件标准，确保最终用户无需专门培训即可轻松完成浏览、注册、购书等操作。
- **管理界面**：后台管理功能同样以桌面窗体形式提供，与前台保持一致的交互逻辑。通过提供清晰的操作按钮和表单，并辅以必要的操作提示或简易手册，书店管理员可以快速上手，完成日常管理工作。
- **结论**：系统设计充分考虑了目标用户（顾客和管理员）的计算机操作水平，**操作可行**。

### 3.4 法律与社会可行性

- **法律合规性**：系统为原创开发，不涉及抄袭或盗用第三方代码、设计，无知识产权侵权风险。项目过程中将注意对所用开源组件的许可协议合规性。
- **社会与伦理**：系统处理用户注册信息，在项目设计中将明确隐私数据的使用范围（仅限于课程演示与系统功能），并在界面提供隐私声明，符合课程伦理与数据安全的基本要求。项目本身促进了文化产品的传播与数字化管理，具有积极意义。
- **结论**：项目在法律和社会层面无负面影响，**法律与社会可行**。

**综合结论**：“网上书店管理系统”项目在技术、经济、操作及法律社会四个方面均具备充分的可行性，风险评估较低，建议项目正式启动。

## 4. CASE工具调研与应用方案

### 4.1 调研概述

为提升软件开发效率与团队协作水平，小组对适用于本项目技术栈和开发流程的计算机辅助软件工程（CASE）工具进行了调研与评估。

### 4.2 选定工具及理由

1. **分析与设计工具**： **Visual Studio 内置建模工具**：作为主要设计工具。其类设计器、架构依赖图等能与C#/C++代码双向同步，实现“设计即代码”，无缝集成，学习成本为零。 **Draw.io / PlantUML**：作为辅助绘图工具。Draw.io用于快速绘制系统架构图、界面原型图；PlantUML通过文本语法生成UML图（如用例图、时序图），生成的文本文件便于使用Git进行版本管理和差异比较，非常适合团队协作更新设计。
2. **版本控制与协同开发工具**： **Git**：选定为分布式版本控制系统。管理所有源代码、文档的版本历史，支持分支策略，是团队并行开发的基础。 **GitHub / Gitee**：选定为远程代码仓库与协作平台。提供代码托管、Issue跟踪（用于任务和Bug管理）、Wiki（项目文档）、Projects（看板式项目管理）等一体化服务。
3. **测试工具**： **Visual Studio Test Explorer**：作为主要单元测试框架。直接集成在IDE中，支持为C++和C#项目创建和运行单元测试，覆盖率高，使用方便。
4. **项目管理工具**： **GitHub Projects / Trello**：利用其看板功能管理项目任务，列如“待办”、“进行中”、“已完成”，可视化工作流，简便灵活。

### 4.3 工具应用整合方案

- **需求与设计阶段**：使用Draw.io绘制系统上下文图、架构图；使用PlantUML编写用例描述和核心业务流程；所有设计文档存入Git仓库。
- **编码实现阶段**： 在Visual Studio中创建解决方案，利用服务器资源管理器进行数据库连接与设计。 采用Git Feature Branch工作流，每个功能在独立分支开发，通过Pull Request（PR）进行代码评审后合并至主分支。 使用GitHub Issues记录和分配开发任务、Bug，并用GitHub Projects看板跟踪整体进度。
- **测试与交付阶段**：在VS中编写并运行单元测试；使用Git Tag标记可交付的版本（如v1.0.0）；最终项目报告、用户手册等纳入Git仓库管理。

## 5. 初步项目计划与团队分工建议

### 5.1 项目阶段划分

1. **第一阶段：需求细化与系统设计（2周）** **主要任务**：细化用例规约，完成数据库概念模型（ER图）与物理设计，绘制系统主要界面原型，确定核心类结构。 **产出物**：详细需求说明文档、数据库设计文档、界面原型图、UML类图（草图）。
2. **第二阶段：系统架构搭建与核心模块开发（3周）** **主要任务**：搭建VS解决方案结构；实现数据访问层（DAL）；使用C++完成核心业务逻辑层（BLL），包括用户、图书、订单等实体类及关键算法。 **产出物**：可运行的核心逻辑模块、初始数据库。
3. **第三阶段：用户界面开发与系统集成（3周）** **主要任务**：使用C# WPF/Windows Forms开发前台客户界面与后台管理界面；实现界面与核心业务逻辑的集成；完成所有功能的初步联通。 **产出物**：具有完整功能的可交互系统原型。
4. **第四阶段：系统测试、调试与交付（2周）** **主要任务**：进行单元测试、集成测试与系统测试；修复Bug；优化用户体验；整理所有项目文档、源代码，撰写用户手册与最终报告。 **产出物**：稳定的可执行程序、完整的项目源代码、测试报告、用户手册、最终项目报告。

### 5.2 团队分工建议

建议采用“模块负责制”与“交叉协作”相结合的方式：

- **核心逻辑组（2人）**：主要负责C++后端业务逻辑、数据访问层及算法实现。
- **前端界面组（2人）**：主要负责C# GUI界面的设计、开发及与后端逻辑的对接。
- **数据库与测试组（1人）**：负责数据库的详细设计、实现、优化，并主导制定测试计划、编写测试用例及执行测试。
- **项目协调与文档（全员参与，指定负责人）**：负责Git仓库维护、进度跟踪、会议记录及主导各类文档的编写与整合。

**协作要求**：所有成员必须熟练使用Git/GitHub进行日常代码提交和协同；定期进行代码评审（PR）；重要设计决策需经小组讨论。
